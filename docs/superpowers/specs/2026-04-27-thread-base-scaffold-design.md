# thread-base ‚Äî Initial Scaffold Design

**Date:** 2026-04-27
**Status:** Approved
**Scope:** Initial scaffold of an empty Vue 3 + TypeScript + Bun + Supabase project. No application features.

## Goals

Create a clean, modern, full-stack-ready Vue project named `thread-base` with:

- Vue 3 + TypeScript + Vite + Bun
- Tailwind CSS v4
- oxlint + Prettier
- Vitest
- Supabase: CLI-based local stack (Docker), `@supabase/supabase-js` client, generated TypeScript types committed to the repo
- Auth **infrastructure** only ‚Äî Supabase client singleton, Pinia auth store, `useAuth()` composable, `requireAuth` router guard. **No login views.**

Future-compatible with Capacitor (mobile apps) ‚Äî design must not preclude that addition.

## Non-Goals

- No login/signup/password-reset views (later)
- No application domain logic, schema, or features (later)
- No Capacitor integration in this scaffold (later, but design accommodates it)
- No CI/CD, no Docker container for the app, no deployment config
- No Storybook, no Playwright

## Stack

| Concern                   | Choice                     | Notes                                                         |
| ------------------------- | -------------------------- | ------------------------------------------------------------- |
| Build tool                | Vite                       | Bun-compatible; standard Vue tooling                          |

| Backend tooling           | Supabase CLI               | Local Docker stack, versioned migrations                      |

**Explicitly excluded (from scaffold):** Nuxt (no SSR needed), `@supabase/auth-ui-vue` (user wants to build UI later), Playwright, Storybook.

## Auth Architecture

Three layers, cleanly separated:

### `src/lib/supabase.ts` ‚Äî Client Singleton

```ts
import { createClient } from '@supabase/supabase-js'
import type { Database } from '@/types/database'

export const supabase = createClient<Database>(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY,
)
```

A single client for the whole app. The `Database` generic types `.from('table')` calls automatically.

### `src/stores/auth.ts` ‚Äî Pinia Store

State:

- `user: User | null`
- `session: Session | null`
- `loading: boolean` ‚Äî true until `initialize()` completes the first time

Actions:

- `initialize()` ‚Äî calls `supabase.auth.getSession()` to populate state, then subscribes to `supabase.auth.onAuthStateChange()` to keep state in sync. Idempotent (safe to call multiple times ‚Äî only subscribes once).
- `signOut()` ‚Äî wraps `supabase.auth.signOut()`.

Getters:

- `isAuthenticated` ‚Äî `!!user && !!session`

### `src/composables/useAuth.ts` ‚Äî Component-facing API

```ts
export function useAuth() {
  const store = useAuthStore()
  return {
    user: computed(() => store.user),
    session: computed(() => store.session),
    isAuthenticated: computed(() => store.isAuthenticated),
    signOut: store.signOut,
  }
}
```

Components import `useAuth()`, never `useAuthStore()` directly. The store is an internal implementation detail; the composable is the public API.

### `src/router/guards.ts` ‚Äî `requireAuth`

- If `auth.loading` is true, await its completion (resolves once `initialize()` finishes).
- If `!auth.isAuthenticated`, redirect to `/login` with `redirect=<originalPath>` query param so post-login navigation can return.
- The `/login` route exists in `router/index.ts` but its component is a stub `<div>Login coming soon</div>` for now.

### `main.ts` ‚Äî Bootstrap Order

```ts
const app = createApp(App)
app.use(createPinia())
const auth = useAuthStore()
await auth.initialize() // critical: BEFORE router + mount
app.use(router)
app.mount('#app')
```

`initialize()` must complete before mount so the first route render sees correct auth state. This avoids flicker and a race condition where `requireAuth` fires before the session is loaded.

## Capacitor Future-Compatibility

Capacitor (mobile apps) will be added later. To avoid painting ourselves into a corner:

- **Routing:** `createWebHistory` for now. When Capacitor is added, switch to `createWebHashHistory` or use Capacitor's deep-link config ‚Äî decision deferred.
- **No Node-only APIs in `src/`** ‚Äî everything must run in browser/WebView. Currently no risk; document as a constraint.
- **Auth redirect URLs:** Mobile will need a custom URL scheme (e.g., `app.threadbase://auth/callback`) instead of `https://`. Today nothing hardcodes `window.location.origin`, so the future change is localized.
- **Asset paths:** Use Vite's `base` config for relative asset paths (default is fine for now).

These are notes, not work items for this scaffold.

## Lint / Format / Test Config

### oxlint ‚Äî `.oxlintrc.json`

```json
{
  "$schema": "./node_modules/oxlint/configuration_schema.json",
  "plugins": ["typescript", "vue", "import"],
  "categories": { "correctness": "error", "suspicious": "warn" },
  "ignorePatterns": ["dist", "node_modules", "src/types/database.ts"]
}
```

Generated DB types are excluded so lint output stays clean.

### Prettier ‚Äî `.prettierrc`

```json
{
  "semi": false,
  "singleQuote": true,
  "printWidth": 100,
  "trailingComma": "all"
}
```

oxlint does not handle formatting, so no `eslint-config-prettier`-style integration is needed.

### Vitest ‚Äî `vitest.config.ts`

- `environment: 'jsdom'` (Vue components testable)
- `globals: true` (no boilerplate imports for `describe` / `it`)
- Plugins: same Vue + tsconfig-paths plugins as the dev config
- One smoke test in `src/App.spec.ts` that mounts `<App />` and asserts no error

### TypeScript

Standard Vue 3 setup with project references:

- `tsconfig.json` (root, only `references`)
- `tsconfig.app.json` ‚Äî app code, `lib: ["ES2022", "DOM"]`, path alias `"@/*": ["./src/*"]`
- `tsconfig.node.json` ‚Äî Vite config and other Node-context files

Path alias is shared with Vite via `vite-tsconfig-paths` (single source of truth).

### Vite ‚Äî `vite.config.ts`

Plugins: `@vitejs/plugin-vue`, `@tailwindcss/vite`, `vite-tsconfig-paths`, `vite-plugin-vue-devtools`.

## CLAUDE.md

A `CLAUDE.md` at the repo root captures project-specific guidance Claude needs when working in this codebase. It is intentionally separate from `README.md` (developer-facing) and the spec (decision log).

Contents:

- **Stack summary** ‚Äî locked-in choices (Vue 3, Bun, oxlint not ESLint, Tailwind v4 not v3, Composition API only)
- **Commands cheat sheet** ‚Äî every `bun run ‚Ä¶` script and what it does
- **Architectural rules** ‚Äî `lib/` for service clients, `composables/` for Vue helpers, `stores/` for Pinia; components use `useAuth()` not `useAuthStore()` directly; auth must be initialized before mount
- **Path alias** ‚Äî `@/*` ‚Üí `src/*`
- **DB workflow** ‚Äî migrations live in `supabase/migrations/`, regenerate types after schema changes
- **Future constraints** ‚Äî Capacitor planned: no Node-only APIs in `src/`, redirect URLs must stay configurable
- **Tooling guardrails** ‚Äî prefer newest stable tools; don't suggest ESLint/npm/yarn/Tailwind v3/Options API

## README

Setup steps:

1. Install Bun (`curl -fsSL https://bun.sh/install | bash`)
2. Install Supabase CLI (`brew install supabase/tap/supabase`)
3. `bun install`
4. `bun run db:start` ‚Äî first run downloads Docker images
5. Copy printed keys into `.env.local`
6. `bun run db:types` ‚Äî generates DB types
7. `bun run dev`
